# PatrickOS v0.4.0-alpha Plan

## Objetivo

Preparar OpenClaw Beta-1 con contratos de herramientas accionables,
sandbox de proceso, filesystem boundaries, confirmación humana y
pruebas negativas antes de cualquier ejecución real. v0.4 sigue
siendo un release de **preparación** — el runtime real no se prende
en este ciclo.

## Enfoque

- Tool contracts accionables: pasar de la spec en
  [`OPENCLAW_TOOL_CONTRACTS.md`](OPENCLAW_TOOL_CONTRACTS.md) a un
  esquema con validador efectivo.
- Allowlist concreta: definir las primeras entradas de
  `configs/openclaw-tools.yaml` como candidatas (no `enabled`).
- Sandbox de proceso: diseño (bwrap / firejail / equivalente
  disponible en la ISO) con mounts mínimos.
- Filesystem boundaries: bind-mounts read-only fuera del workspace,
  sin acceso a `$HOME` (fuera de `.patrick-os/`), `/etc`, `/tmp`
  global.
- Confirmación humana obligatoria por step (no global).
- Ejecución simulada avanzada antes de ejecución real: extender el
  `blocked-by-design` con "simulación" — qué pasaría si el runtime
  corriera, registrado en plan + audit, sin tocar el FS real.
- Más pruebas negativas: ver
  [`OPENCLAW_NEGATIVE_TESTS.md`](OPENCLAW_NEGATIVE_TESTS.md).
- Mantener Watson como supervisor: ningún path agéntico habla con
  el usuario directo.

## Scope explícito

### En scope

- `docs/OPENCLAW_BETA1_PLAN.md` (fases + candidatas + reglas).
- `docs/OPENCLAW_NEGATIVE_TESTS.md` (qué tests negativos exigir
  antes de prender la primera herramienta).
- Extensión del execution gate hacia ejecución simulada
  (`blocked-by-design` enriquecido con un "would run" detallado,
  igual sin ejecutar).
- Validador efectivo del tool registry (no solo `tools=[]`).
- Diseño de sandbox documentado (sin todavía implementarlo en
  código de runtime).

### Fuera de scope (explícito)

- Ejecución real de herramientas.
- `sudo` desde agentes.
- Red habilitada para OpenClaw.
- Plugins externos.
- Marketplace.
- Daemon agéntico o transporte TCP.
- NVIDIA propietario.
- Firefox / LibreOffice / OBS preinstalados.
- ISO `v0.4` — no se construye en este ciclo.

## Validación por PR

Cada PR de v0.4 debe pasar:

- `make check` → `fails=0`.
- `watson policy check` → 7 invariantes OK (incluyendo
  `tool registry: disabled/empty`).
- `watson tools list` → `"No hay herramientas habilitadas."`
- `watson claw execute --mode <m> <plan-aprobado>` → siempre
  `Estado: blocked-by-design`.
- `watson readiness` → `ready_for_simulated_beta1=yes`,
  `ready_for_real_execution=no` (`[BLOCKED] real execution
  runtime not implemented` es estado por diseño).
- Sin cambios en `iso/` salvo PR explícitamente etiquetado.

## Criterio para empezar Beta-1

Beta-1 (primera herramienta habilitada) **solo puede empezar
cuando todas estas precondiciones se cumplan**:

- `v0.3.0-alpha` está publicado (tag + ISO + release notes).
- OpenClaw Beta-0 está cerrado según
  [`OPENCLAW_BETA0_CHECKLIST.md`](OPENCLAW_BETA0_CHECKLIST.md).
- Tool contracts existen y un validador efectivo los aplica.
- Allowlist está definida (entradas en `openclaw-tools.yaml` con
  `state` por entrada, todavía no `enabled`).
- Sandbox está diseñado (doc + decisión de tecnología).
- `watson doctor` sigue `FAIL=0` después de cada cambio.
- Pruebas negativas listadas en
  [`OPENCLAW_NEGATIVE_TESTS.md`](OPENCLAW_NEGATIVE_TESTS.md) están
  identificadas, incluso si no todas son ejecutables todavía.

Si falta cualquiera, no se abre Beta-1. La primera herramienta
habilitada entra en un PR aparte con su propia revisión de
seguridad.
